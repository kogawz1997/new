Trial Flow

What to verify before go-live

This flow assumes the platform keeps owner, tenant, and player experiences separated while sharing the same operational runtime.

1. Access

Owner and tenant routing

Confirm global users land in the owner console and tenant-scoped admins land in the tenant console without overlap.

2. Runtime

Bot, worker, watcher, and optional agents

Validate runtime health, service readiness, and alert visibility from the owner console before tenant onboarding.

3. Tenant ops

Catalog, delivery, and wallet workflows

Run tenant-scoped config, queue, wallet, and delivery recovery flows without touching owner-only controls.

4. Player trust

Wallet, purchase, redeem, and Steam link

Walk through the player portal on mobile and desktop to confirm a clean, understandable transaction experience.

Owner Checklist

Before opening tenants

Runtime health looks stable

Owner overview shows managed services and agents in a known-good state.

Security and audit review is available

Security events, request anomalies, and owner-only governance links are visible from one console.

Global config and recovery are still guarded

High-risk flows remain in explicit owner-only workbench areas with confirmation steps.

Tenant + Player Checklist

Before opening the server community

Tenant catalog and delivery are ready

Shop items, delivery queue, dead letters, and safe recovery actions are visible in the tenant console.

Player portal is understandable

Wallet state, item purchase flow, redeem, and Steam-link requirements are explicit to the player.

Legacy surfaces remain reachable

Older workflows stay reachable for migration and operator depth without breaking the new primary UX.
